## 개요
[**CodeQL**](https://codeql.github.com): Semmle이 개발하고 GitHub이 2019년에 인수한 Semantic Code Analysis Engine

- Semantic Code Analysis Engine(의미론적 코드 분석 엔진): 코드가 실행되는 의미와 의도를 분석하는 기술

**핵심 아이디어**: 코드를 데이터인 것처럼 쿼리하여 취약점을 분석한다. 

**기존 정적 분석과의 차이**

| 구분      | 기존 정적 분석       | CodeQL                            |
| --------- | -------------------- | --------------------------------- |
| 분석 방식 | 패턴 매칭, 규칙 기반 | 시맨틱 분석(의미 분석)            |
| 표현력    | 단순 패턴만 가능     | 복잡한 데이터/제어 흐름 표현 가능 |
| 오탐율    | 상대적으로 높음      | 상대적으로 낮음                   |
| 분석 범위 | 단일 파일/함수       | 프로젝트 전체 흐름 추적 가능      |

**구성 요소**
1. CodeQL Database - 소스 코드를 분석 가능한 형태로 변환한 관계형 DB
2. Query Language - CodeQL 전용 쿼리 언어
3. CodeQL CLI / Extensions - 분석 실행 환경

**분석에 포함되는 3가지 흐름**
1. 데이터 흐름 - 데이터가 어디에서 어디로 이동하는지 추적
2. 제어 흐름 - 코드 실행 경로에 따른 조건 분석
3. 호출 그래프 - 함수간 호출 관계를 추적 및 분석

## 동작 원리
**전체 흐름**  
```
소스 코드  
    ↓ ① Extraction (추출)  
CodeQL Database  
    ↓ ② Query Execution (쿼리 실행)  
Raw Results  
    ↓ ③ Post-processing (후처리)  
최종 분석 결과 (경고, 취약점 위치)
```

---

### ① Extraction (추출)   
소스 코드 데이터를 추출하여 DB에 맞게 변환 후 저장하는 단계이다. 

**추출 과정**
```
소스 코드(.java, .py, .js ...)
    ↓
[언어별 Extractor]
    ↓
① AST (추상 구문 트리)       → 코드 문법 구조
② CFG (제어 흐름 그래프)     → 실행 경로
③ DFG (데이터 흐름 그래프)    → 값의 이동 경로
④ Call Graph (호출 그래프)   → 함수 간 관계
    ↓
CodeQL Database (.db)
```

**AST**: 코드를 트리 구조로 분해한 데이터 구조

### ② Query Execution (쿼리 실행)
준비된 DB에서 쿼리를 실행하여 취약점을 찾는 단계이다. 
- QL은 SQL과 유사하지만 코드 의미를 표현할 수 있다. 

---

### ③ Post-processing (후처리) 
Raw Results를 사람이 읽기 좋은 형태로 변환하고 의미를 해석한다. 
- 중복 제거
- 심각도 (Severity) 분류
- 위치 정보 매핑
- 취약점 경로 시각화 

## 한계점
- **느린 실행 속도** - 실습을 진행하면서 느낀 점이 속도가 굉장히 느리다는 것이다. 간단한 코드임에도 이를 변환하여 DB로 저장해야 하기 때문에 시간이 오래 걸린다. 
- **언어와 프레임워크 한계** - 지원 언어 외에 분석이 불가하고, 커스텀 함수나 지원하지 않는 프레임워크는 직접 모델링을 해줘야 한다. 
- **QL 언어 숙지 필수** - QL 문법을 별도로 공부하지 않으면 쿼리를 작성할 수 없다. 
- **동적 분석 불가** - 정적 분석 모델이기 때문에 런타임 동작은 분석하지 못한다. 
- **런타임 동작**: 작성된 프로그램이 실제로 작동되는 동안 발생하는 일
- **오탐 발생 가능성** - 실제로 위험한 코드를 놓치거나, 안전한 코드를 위험하다 판단하는 오류가 발생할 수 있다.

### 5. QL 쿼리 작성 & 실행
**쿼리 작성 경로 및 파일 생성**  
`mkdir queries`  
`touch queries/find-sqli.ql`

**QL 코드 작성**
```
/**
 * @name SQL Injection + XSS 탐지
 * @description 사용자 입력이 SQL 쿼리 또는 HTML 응답으로 흘러가는 경로 탐지
 * @kind path-problem
 * @severity error
 * @id java/sqli-xss-custom
 */

import java
import semmle.code.java.dataflow.TaintTracking
import semmle.code.java.dataflow.FlowSources

private module SqlInjConfig implements DataFlow::ConfigSig {
    predicate isSource(DataFlow::Node source) {
        source instanceof RemoteFlowSource
    }

    predicate isSink(DataFlow::Node sink) {
        // SQL Injection Sink: java.sql 의 execute 계열 메서드
        exists(MethodCall mc |
            mc.getMethod().getName().matches("%execute%") and
            mc.getMethod().getDeclaringType().hasQualifiedName("java.sql", _) and
            sink.asExpr() = mc.getAnArgument()
        )
        or
        // XSS Sink: HttpServletResponse.getWriter().print/println/write 계열 메서드
        exists(MethodCall mc |
            mc.getMethod().getName().matches("%print%") and
            mc.getMethod().getDeclaringType().hasQualifiedName("java.io", "PrintWriter") and
            sink.asExpr() = mc.getAnArgument()
        )
    }
}

module VulnFlow = TaintTracking::Global<SqlInjConfig>;
import VulnFlow::PathGraph

from VulnFlow::PathNode source, VulnFlow::PathNode sink
where VulnFlow::flowPath(source, sink)
select sink.getNode(), source, sink,
    "취약점 발견: 사용자 입력이 검증 없이 위험한 지점으로 전달됨"
```
쿼리 파일을 우클릭하여 "CodeQL: Run Query on Selected Database"로 실행

## 소감
&nbsp; "코드를 데이터베이스에 쿼리로 보낸다"는 발상이 굉장히 신선하게 다가왔다. 데이터 흐름, 제어 흐름, 호출 그래프를 통해 보다 종합적으로 취약점을 분석하는 점이 실용적이라고 느껴졌다. 또한 SQL 언어와 유사한 점이 많아 익숙하게 느껴지기도 했다. 이러한 도구를 잘 활용하는 것이 굉장히 중요하리라 생각되어 앞으로 더 깊게 공부해보고 싶은 생각이 들었다.
